=== Key Architectural Differences ===

Unfused:
  - 3 separate phases: QK^T -> Softmax (via NMP) -> V multiply
  - All intermediates (raw scores + attention weights) written to SRAM
  - Simplest FSM, lowest area

Partial Fused:
  - 2 phases: Fused QK^T+Softmax -> V multiply
  - Online softmax (running max/sum) keeps raw scores in registers
  - Attention weights still written to SRAM for V multiply phase
  - Required addtree pragma fix for SCHD-3 (running_sum feedback path)

Fully Fused:
  - 1 phase: Fused QK^T+Softmax+V multiply (FlashAttention-style)
  - Online softmax with correction factors applied to V accumulator
  - All intermediates in registers, only final output written to SRAM
  - 2-thread dataflow split (MemCtrlRun + ComputeRun) needed for HLS
  - Required addtree pragma fix for SCHD-3
  - 2x area cost for full fusion

=== HLS Pragmas Used ===

Unfused + Partial Fused:
  #pragma hls_pipeline_init_interval 3    (main while loop, II=3)

Fully Fused:
  MemCtrlRun:  #pragma hls_pipeline_init_interval 1  (II=1)
  ComputeRun:  #pragma hls_pipeline_init_interval 3  (II=3)

All variants:
  #pragma hls_unroll yes                  (vector loops, 16 iterations)

Partial + Fully Fused:
  #pragma cluster addtree                 (running_sum accumulation)
  #pragma cluster_type both               (addtree for both signed/unsigned)

Fully Fused only:
  ac_channel<> for MemCtrl<->Compute      (2-thread dataflow)
  Connections::Combinational<> channels   (inter-thread communication)

=== Notes ===

- SCHD-3 "Feedback path too long" was the main HLS challenge
  - Caused by 16-deep sequential addition chain on running_sum through large FSM MUX
  - Fix: local tile_sum variable with addtree pragma reduces depth to O(log2(16))=4
  - Affected both Partial Fused and Fully Fused variants
- Fully Fused additionally required 2-thread dataflow split to separate
  memory control from compute logic
- N=16 differences are modest because intermediate matrices are small (16x16)
- N=64 is where fusion truly matters: N^2 intermediates dominate traffic
- All variants use same fixed-point types: int8 I/O, 32-bit intermediates, 40-bit accumulators
- Cycle counter bug fix: original counter incremented during IDLE state (counting
  wait loops). Fixed: if (state != IDLE) attn_config.cycle_count++
- IMPORTANT: Internal cycle_count is NOT a reliable cross-variant metric.
  cycle_count counts FSM iterations (one per wait()), NOT real clock cycles.
  Variants have different IIs and blocking channel stalls add hidden cycles.
  Following solution-lab-4's approach: use scverify simulation timestamps
  (sc_time_stamp() at DONE - start) as the ground truth for sim clocks.
  Solution-lab-4's NMP/PECore have NO internal cycle counters — only sc_time_stamp().
- Sim clocks (from scverify timestamps, tb.clk = 1ns):
  Unfused: 84,296 | Partial Fused: 131,541 | Fully Fused: 66,454
  Speedup: Fully Fused is 1.27x faster than Unfused for N=64
- Partial Fused is slowest due to RTL PopNB stalls — write/read interleaving
  on the same GBCore memory channel adds ~2 stall cycles per memory read
- Key metric is GB writes (proxy for off-chip traffic), not cycles:
  Unfused=576, Partial=320, Fused=64 — fusion eliminates intermediate materialization
- Fusion tradeoff: Fully Fused wins on latency AND data movement, but costs 2x area

Note: FPGA is slower than CPU at these small sizes. This is expected:
- CPU runs at ~3+ GHz vs FPGA at 125 MHz (24x clock advantage)
- N=16-64 fits entirely in CPU L1 cache (~10 KB working set)
- FPGA advantage emerges at larger N where off-chip memory bandwidth
  becomes the bottleneck (N^2 intermediate matrices exceed cache/SRAM)
- The key result is the relative comparison between fusion variants,
  not absolute FPGA vs CPU speedup at toy sizes
